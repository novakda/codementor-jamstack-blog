**Step 1 — Overview:** Before any file creation, present a brief overview of all 5 steps the user will complete in Phase 1:

2. CMS-02: Create admin/config.yml — tells the CMS which backend to use

Explain what Decap CMS is: a single-page React application that provides a browser-based content editor for markdown files. It was formerly known as Netlify CMS (renamed February 2023). For Phase 1, we use a "test-repo" backend — an ephemeral in-memory backend that lets us test the CMS interface without needing Git authentication. Data won't persist across page reloads. Phase 2 will switch to a real backend.

**Step 2 — Create index.html:** Instruct the user to:
1. Create a new folder: `src/admin/`
2. Create `src/admin/index.html` with this exact content:

```html
<!doctype html>
<html>
<head>
  <meta charset="utf-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0" />
  <meta name="robots" content="noindex" />
  <title>Content Manager</title>
</head>
<body>
  <script src="https://unpkg.com/decap-cms@^3.0.0/dist/decap-cms.js"></script>
</body>
</html>
```

**Explain WHY each part matters:**
- `noindex` meta tag: Prevents search engines from indexing your admin panel
- The script tag loads the entire CMS application from a CDN (no npm install needed)
- `^3.0.0` means "any version 3.x" — gets patches but avoids breaking changes
- The CMS automatically looks for `config.yml` in the same folder as this HTML file

**Walkthrough delivery:** One step at a time, explain what AND why. Present config.yml in sections so the user understands each part.

**Step 1 — Backend configuration (CMS-02):** Explain what a "backend" is in CMS terms: it's how the CMS reads and writes content. Different backends connect to different storage systems.

Instruct user to create `src/admin/config.yml` starting with:

```yaml
backend:
  name: test-repo
```

Explain: `test-repo` is an ephemeral in-memory backend for local testing. It won't show existing blog posts and data disappears on page reload. This is intentional — we're testing the CMS interface works correctly. Phase 2 switches to `git-gateway` for real persistence.

**Step 2 — Media paths (CMS-04):** Add media configuration:

```yaml
media_folder: "src/assets/blog"
public_folder: "/assets/blog"
```

Explain the difference:
- `media_folder`: Where uploaded files are physically saved in the repository (`src/assets/blog/`)
- `public_folder`: The URL path written to frontmatter (`/assets/blog/image.jpg`)
- These differ because Eleventy transforms `src/` → `public/` during build, stripping the `src/` prefix
- Existing posts already use `/assets/blog/article-1.jpg` — CMS uploads must match this pattern

**Step 3 — Collection and fields (CMS-02, CMS-03):** Add the blog collection:

```yaml
collections:
  - name: "blog"
    label: "Blog"
    folder: "src/blog"
    create: true
    slug: "{{year}}-{{month}}-{{day}}-{{slug}}"
    fields:
      - {label: "Title", name: "title", widget: "string", required: true}
      - {label: "Author", name: "author", widget: "string", default: "Kevin Powell", required: false}
      - {label: "Date", name: "date", widget: "datetime", format: "YYYY-MM-DD", date_format: "YYYY-MM-DD", time_format: false, required: true}
      - {label: "Featured", name: "featured", widget: "boolean", default: false, required: false}
      - {label: "Image", name: "image", widget: "image", required: false}
      - {label: "Image Alt Text", name: "imageAlt", widget: "string", required: false}
      - {label: "Description", name: "description", widget: "text", required: false}
      - {label: "Body", name: "body", widget: "markdown"}
```

Explain each section:
- `name`/`label`: Internal ID vs display name in CMS sidebar
- `folder: "src/blog"`: Where markdown files live — matches your existing blog posts location
- `create: true`: Allows creating new posts (not just editing existing ones)
- `slug: "{{year}}-{{month}}-{{day}}-{{slug}}"`: Generates filenames like `2026-03-04-my-post-title.md` — matches existing naming convention

Explain the field choices and WHY:
- **title** (string, required): Every post needs a title — maps to `title:` in frontmatter
- **author** (string, default "Kevin Powell"): Pre-fills but editable — saves typing for the common case
- **date** (datetime): `format: "YYYY-MM-DD"` outputs dates like `2021-05-01` matching existing posts. `time_format: false` hides the time picker since we only need dates
- **featured** (boolean): Toggle switch that outputs `featured: true/false`. NOTE: existing posts use `tags: ["featured"]` — template will need to handle both patterns
- **image** (image): Upload widget with media library browser. Uses the media_folder/public_folder paths configured above
- **imageAlt** (string): Separate text field for accessibility — paired with image
- **description** (text): Multi-line plain text for post summaries. `text` widget gives a larger textarea than `string`
- **body** (markdown): Rich text WYSIWYG editor — the main content area with formatting toolbar

Explain what's NOT in the config and why:
- No "tags" field for "post" — blog.json auto-applies this to every post in src/blog/
- No "layout" field — blog.json auto-applies "article.njk" layout
- Only title and date are required — minimum viable post for quick drafting
